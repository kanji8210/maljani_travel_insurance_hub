Maljani Travel Insurance Hub — Support Chat Functionality

Overview
- This plugin provides a support chat system integrated into WordPress for public users and site admins.
- Conversations are stored in the database as sessions and messages; admin UI provides a WhatsApp-like conversation view.

Key Components
1) Database
- Tables (created by DB tools or scripts):
  - wp_support_sessions: session per conversation (id, user_id, email, subject, status, last_message_at)
  - wp_support_messages: message entries linked to a session (id, session_id, sender[user|agent], user_id, email, message, status, created_at)
  - wp_support_email_queue: queued outgoing emails (schema present; plugin currently uses an option-backed queue but DB table exists for migration)

2) REST API
- POST /wp-json/support-chat/v1/send
  - Public endpoint for users to send messages; finds/creates a session and inserts a user message.
- GET /wp-json/support-chat/v1/unread-count
  - Returns unread count (admin/editor-only).
- GET /wp-json/support-chat/v1/session/{id}[?after_id=NN]
  - Returns all messages (or messages after id) for a session (admin-only).
- POST /wp-json/support-chat/v1/session/{id}/message
  - Admin POST endpoint to insert an agent message into a session (admin-only).

3) Admin UI
- Admin menu: "Support Chat" shows recent sessions with per-session unread badges.
- Session view: conversation bubbles, timestamps, reply box.
- Optimistic UI: agent replies are appended immediately (pending state) and reconciled after REST response.
- Admin scripts poll unread count (30s) and session messages (5s) when viewing a session. If a WebSocket server is configured, the admin connects to receive real-time broadcasts.

4) WebSocket Scaffold
- scripts/ws-server/: node-based WebSocket + HTTP broadcast server (server.js).
- POST /broadcast on the WS server accepts JSON payloads from plugin (notify_ws_server helper) and broadcasts to connected admin clients.
- The WS server is optional; plugin falls back to polling if unavailable.

5) Emailing & Queue
- Template options exist for new-message and response emails.
- Plugin queues outgoing emails using an option-backed queue and a scheduled processor; DB-backed queue table exists for migration.

6) Helper Scripts (scripts/)
- create_support_conversation_tables.php — creates support_sessions and support_messages tables from CLI.
- simulate_session_send.php — CLI simulate user send (creates session + message).
- simulate_agent_reply.php — CLI simulate agent reply into a session.
- ws-server/ — WebSocket server scaffold with README and npm scripts.

Testing & Running
- Create tables: run the DB Tools page or run: php scripts/create_support_conversation_tables.php
- Simulate a user message: php scripts/simulate_session_send.php email@example.com "Hello"
- Simulate agent reply: php scripts/simulate_agent_reply.php SESSION_ID "Reply text"
- Start WS server (optional):
    cd scripts/ws-server
    npm install
    npm start
- Verify admin view: Login to WP admin → Support Chat → open session; messages should appear and updates arrive via WS or polling.

Notes & Next Steps
- Recommended: migrate the option-backed email queue to the DB-backed `wp_support_email_queue` and add a manual queue processor in admin.
- For production WS: secure the server (TLS), add authentication, and configure plugin options `maljani_ws_server_url` and `maljani_ws_server_http`.
- Additional UX: retry button for failed optimistic messages, richer styling and mobile responsiveness.
